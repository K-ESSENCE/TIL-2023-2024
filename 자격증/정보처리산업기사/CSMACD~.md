### CSMACD

CSMA/CD 방식

CSMA 방식에서와 같은 충돌이 발생하는 문제점을 해소하기위해 CSMA 방식에 충돌 검출 기능과 충돌 발생 시 재송신하는 기능을 부가

통신하기전에 누가 이 통신회선 쓰고있는지 확인

Carrier Sense => 통신회선이 다른애들이쓰고있는지 아닌지점검하는것

Multiple Access => 여러개가 액세스하면 충돌이 발생하면 데이터가 정상적으로 전달이 안됨

Collision Detection => 충돌여부 확인

이런문제점을 해결하기위해 CSMA에 충돌검출과 충돌시 재송신하는 기능부가한것 => CMSACD.

CSMA CD방식은 통신회선이 사용중이면 대기하고, 데이터가 없을떄에만 데이터를 송신.

송신중에도 전송로의 상태를 계속 감시.

충돌이 발생하면 송신을 중지 후 모든 노드에 충돌을 알린 후 일정 시간이 지난 다음 데이터를 재송신함.

성형 또는 버스형 LAN에서 가장 일반적으로 사용됨.

전송량이 적을떄 매우 효율적이며 신뢰성이 높다

### 소프트웨어 개발방법론의 개요

개발, 유지보수 등에 필요한 수행방법과 과정에 필요한 각종 기법및 도구를 체계적으로 정리해서 표준화한것.

최종 목적은 소프트웨어의 생산성과 품질 향상

구조적 방법론 / 객체지향/ 컴포넌트기반 / 애자일 / 제품 게열 / 정보공학 방법론

### 구조적 방법론

계산기 프로그램 만든다고한다면 순서대로 입력하는것을 절차적 순차적 설계라고 불렀음.
요구사항을 파악하여 문서화하는 처리 중심의방법론이 구조적 방법론

기능을 전부 부품화 / 모듈화 시켜서 필요한애들을가져다 쓰는게 구조적 방법론

복잡한 문제를 다루기 위해 분할과 정복 원리를 적용.

타당성 검토 -> 계획 -> 요구사항 -> 설계 -> 구현 -> 시험 -> 운영 /유지보수

### 정보공학 방법론

정보 시스템의 개발을 위해 계획/ 분석 /설계/ 구축에 정형화된 기법들을 상호 연관성 있게 통합 및 적용하는 자료 중심의 방법론

**자료중심**

대규모 정보 시스템을 구축하는데 적합.

실제 업무 위주로 개발하자 하고 만들어놓은게 정보공학 방법론

정보전략 계획 수립 -> 업무 영역 분석 -> 업무 시스템 설계 -> 업무 시스템 구축

### 객체지향 방법론

현실 세계의 개체(Entity)를 기계의 부품처럼 하나의 객체로 만들어서
소프트웨어를 개발 할 때 기계의 부품을 조립하듯이 객체들을 조립해서 구현하는 방법론

객체지향 방법론의 구성요소: 객체/클래스/메시지
메시지? => 객체들간의 상호작용에 사용되는 수단

객체지향 기본원칙: 캡슐화/정보은닉/추상화/상속성/다형성

캡슐화 -> 하나로 묶는거
정보은닉 -> 캡슐화에서 가장 중요한 개념으로 자신의 정보를 숨기고 자신의 연산만을 통해서 접근허용하는것
추상화 -> 불필요한 부분을 다 생략하고 가장 중요한것에 중점을갖고 개략화하는것
상속성 -> 상위에있는 애를 하위가 물려받는것
다형성 -> 하나로 다양하게 표현하는거

예시를 들면 자동차는 클래스 트럭 승용차 인스턴스
바퀴 문 이런건 속성
전진 후진은 메서드

공통된 특성 => 클래스

분석->설계->구현->테스트->인도

### 스크럼의 개요

럭비에서 반칙으로 경기 중단되면 가운데에 럭비공 두고 양팀이 서로 밀치기위한 대형 (? 어쩔)
스크럼은 이처럼 팀이 중심이 되어 개발의 효율성을 높인다는 의미가 내장되어있다 (아하)

스크럼은 팀원 스스로가 스크럼 팀을 구성해야되고 개발 작업에 관한 모든것을 스스로 해결 할 수 있어야된다.

스크럼 팀은 제품 책임자 / 스크럼 마스터 / 개발팀으로 구성됨

제품 책임자: 개발 의뢰자 | 실제 사용할 담당자 | 가장 이해도가 높은사람

마스터 : 객관적인 시각에서 개발팀에게 조언해주고 가이드 해주는사람 , 팀원 통제가 목표가아니라 일을 잘할수있도록 서포트해주는 역할
개발팀과 제품 책임자 사이의 중간 위치

개발팀 : 책임자와 마스터를 제낀 나머지가 다 개발팀으로 분류 여기안에는 디자이너등도 개발팀으로 포함

백로그는 아무나 다 적을 수 있음

제품 책임자만 우선순위를 지정할 수 있음

개발팀마다 너는 뭐하고 언제까지 뭐해라 이걸 지정할수있는게 책임자

그럼 개별적으로 백로그가 생기는데 스프린트 백로그 라고 불리기도함

개발 시작하면 끝날때까지 매일 스크럼 마스터가 일일스크럼 회의라는것을 주관함

제품 책임자 => P.O => 요구사항을 책임지고 의사결정을 책임질 사람ㅋㅋㅋ

이해관계자들의 의견을 종합해서 제품에 대한 요구사항을 작성하는 주체

백로그는 아무나 추가할수는있으나 우선순위는 지정 불가

스크럼 마스터는 SM 이라고 부름
개발팀은 DT
